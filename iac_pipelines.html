IaC CI/CD Pipelines

인프라 변경을 자동화하고 안전하게 검토하는 파이프라인 구축

1. 인프라 배포의 자동화 흐름

코드 한 줄로 인프라가 바뀌는 만큼, 매우 엄격한 검증 과정이 필요합니다.

  1. Validation: 테라폼 코드의 문법과 정책 준수 여부를 검사합니다. (`terraform validate`, `tflint`)
  2. Plan & Review: 실제 인프라에 어떤 변화가 생길지 리포트를 생성합니다. (`terraform plan`)
  3. Manual Approval: 생성된 리포트를 사람이 확인하고 승인합니다.
  4. Apply: 승인된 변경 사항을 실제 클라우드에 반영합니다. (`terraform apply`)

Pull Request 기반 테라폼 운영 (Atlantis)

GitHub의 Pull Request 댓글을 통해 테라폼 명령을 실행하고, 동료의 코드 리뷰와 승인을 강제하는 방식이 협업에서 널리 쓰입니다.

2. 파이프라인 구성 예시

# GitHub Actions를 이용한 IaC 파이프라인
jobs:
  terraform-plan:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: hashicorp/setup-terraform@v2
      
      - name: Terraform Plan
        run: terraform plan -out=tfplan
        
      - name: Comment Plan to PR # PR 댓글로 변동 내역 자동 게시
        uses: robampersand/terraform-pr-commenter@v1
        with:
          plan: ${{ steps.plan.outputs.stdout }}

실전 팁: 보안 스캐닝 추가

인프라를 배포하기 전에 `tfsec`이나 `Checkov` 같은 도구를 파이프라인에 추가하세요. "S3 버킷이 공개되어 있음" 같은 보안 취약점을 배포 전에 자동으로 찾아낼 수 있습니다.
