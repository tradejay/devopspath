Logging & Correlation (로그 통합)

파편화된 로그를 하나로 모으고 지표와 연결하여 맥락 파악하기

1. 분산 환경에서의 로그 관리

서버가 여러 대이거나 마이크로서비스 아키텍처(MSA)인 경우, 각각의 서버에 들어가서 로그 파일을 확인하는 것은 불가능합니다. 로그를 중앙으로 수집해야 합니다.

Grafana Loki: 지표처럼 다루는 로그

전통적인 ELK(Elasticsearch) 스택과 달리, Loki는 로그 자체의 전문 검색보다는 '메타데이터 인덱싱'에 집중합니다. 덕분에 저장 비용이 매우 저렴하며, Grafana에서 지표와 로그를 한눈에 볼 수 있는 강력한 통합 환경을 제공합니다.

2. 로그 보존 및 검색 전략

  • Retention (보관 주기): 모든 로그를 영원히 저장할 수는 없습니다. 비즈니스 중요도에 따라 보관 주기를 설정하세요.
  • JSON Logging: 텍스트 기반 로그보다는 기계가 읽기 좋은 JSON 형식을 권장합니다. 필드별 검색이 용이해집니다.
  • Contextual Logging: 에러 로그 발생 시 유저 ID, 요청 ID, 환경 정보 등을 함께 남겨야 맥락을 파악할 수 있습니다.
# JSON 로그 형식 예시
{
  "timestamp": "2026-04-21T10:00:00Z",
  "level": "ERROR",
  "service": "order-service",
  "request_id": "req-12345",
  "user_id": "user-88",
  "msg": "Database connection failed",
  "stacktrace": "..."
}

핵심: Correlation ID (상관관계 ID)

특정 요청이 여러 서비스를 거칠 때 동일한 `request_id`를 유지하도록 전파(Propagation)하세요. 그래야 지표에서 발견된 에러 포인트에서 바로 해당 요청의 모든 로그를 필터링할 수 있습니다.
