Logging & Events: 상관관계 분석

단순한 텍스트 로그를 넘어 기계가 읽을 수 있는 데이터로

1. 구조화 로깅 (Structured Logging)

전통적인 "Server started" 같은 텍스트 로그는 사람이 읽기 좋지만, 수백만 줄의 로그에서 특정 거래를 검색하기에는 적합하지 않습니다. 모든 로그를 **JSON 포맷**으로 남기는 것이 실무의 표준입니다. 

// Bad: 텍스트 로그
"User jay ordered item 123 at 14:00"

// Good: 구조화 로그 (JSON)
{
  "timestamp": "2026-04-19T14:00:00Z",
  "level": "INFO",
  "user_id": "jay",
  "event": "ORDER_CREATED",
  "item_id": 123,
  "trace_id": "abc-123-xyz" // 상관관계의 핵심!
}

2. 로그와 메트릭의 상관관계 (Correlation)

메트릭 대시보드(Grafana)에서 500 에러 기둥이 속 솟았다면, 우리는 즉시 **그 시점에 발생한 로그**를 찾아내야 합니다. 로그에 `trace_id`나 `request_id`를 함께 남기면, 특정 지표의 이상 현상과 실제 에러 로그를 1초 만에 연결(Correlation)할 수 있습니다.

3. 실무 팁: 로그 레벨 관리

운영 환경에서는 `DEBUG` 로그를 끄고 `INFO` 이상의 로그만 남겨야 합니다. 너무 많은 로그는 스토리지 비용을 폭증시키고 검색 속도를 늦추기 때문입니다. 장애가 발생했을 때만 동적으로 로그 레벨을 바꿀 수 있는 기능을 구현하는 것이 고수 개발자의 비결입니다.
