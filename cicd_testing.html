Testing Strategy

테스트 피라미드와 자동화된 수락 테스트(Acceptance Testing)

1. 테스트 피라미드 (The Testing Pyramid)

모든 테스트를 동일한 비중으로 작성하면 유지보수 비용이 기하급수적으로 늘어납니다. 전문가들은 다음과 같은 **'피라미드 구조'**를 권장합니다.

  • Unit Tests (수천 개): 아주 빠르고 작은 단위 검증. 전체의 70% 이상 차지.
  • Service/Integration Tests (수백 개): API 연동이나 DB 연결성 검증.
  • UI/Acceptance Tests (수십 개): 사용자 관점에서의 전체 시나리오 검증.

Acceptance Testing: 비즈니스 가치의 증명

단위 테스트가 '코드가 맞게 작성되었나'를 확인한다면, 수락 테스트는 '사용자가 원하는 대로 동작하나'를 확인합니다. 이 단계가 통과되어야만 비로소 '배포 가능한 상태'가 됩니다.

2. 배포 파이프라인에서의 테스트 흐름

테스트는 피드백 루프의 핵심입니다. 실패할 가능성이 높고 빠른 테스트(Unit)를 앞에 배치하고, 비싸고 느린 테스트(E2E/Acceptance)를 뒤에 배치하여 효율적인 파이프라인을 구축합니다. 

# 요약된 테스트 워크플로우 예시
      - name: Run Unit Tests
        run: npm run test:unit
        
      - name: Build and Setup Environment
        run: npm run build
        
      - name: Run Acceptance Tests (e.g. Playwright)
        run: npm run test:e2e

3. 테스트 코드 작성 시 유의사항

  • Flaky Test 제거: 간헐적으로 실패하는 테스트는 자동화의 신뢰도를 떨어뜨립니다. 즉시 수정하거나 격리해야 합니다.
  • Mocking 전략: 외부 API나 복잡한 의존성은 적절히 모킹(Mock)하여 테스트 속도를 유지합니다.
