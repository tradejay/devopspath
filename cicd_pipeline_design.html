Deployment Pipeline 설계

배포 프로세스의 아키텍처와 Pipeline as Code

1. 배포 파이프라인의 해부 (Anatomy)

배포 파이프라인은 소프트웨어가 체크인되는 순간부터 사용자 손에 들어가는 순간까지를 관리하는 **'자동화된 가치 흐름'**입니다. 전문적인 파이프라인은 보통 다음과 같은 스테이지로 구성됩니다.

  • Commit Stage: 빌드, 린트, 단위 테스트 실행.
  • Automated Acceptance Stage: 수락 테스트 실행 및 환경 구성 검증.
  • Manual Approval Stage: 운영 환경 배포 전 사람의 최종 검토 및 승인.
  • Production Deployment: 실제 서비스 환경에 안전하게 배포.

Pipeline as Code (PaC)

배포 파이프라인 설정 자체를 코드로 관리하는 것입니다. Jenkinsfile이나 GitHub Actions의 YAML 파일이 예시입니다. 파이프라인도 버전 관리가 되어야 하며, 변경 이력이 투명하게 남아야 합니다.

2. 스테이지 간 아티팩트(Artifact) 관리

파이프라인의 매 단계마다 새로 빌드하는 것은 위험합니다. 첫 단계에서 빌드된 결과물(Docker 이미지, 바이너리 등)을 **'아티팩트'**로 저장하고, 이후 모든 스테이지는 동일한 아티팩트를 사용하여 '동일한 코드가 테스트되었음'을 보장해야 합니다. 

# 아티팩트 활용 예시 (GitHub Actions)
    steps:
      - name: Build and Upload Artifact
        uses: actions/upload-artifact@v4
        with:
          path: build/
          
    # 다음 Job에서 가져오기
    steps:
      - name: Download Artifact
        uses: actions/download-artifact@v4

3. 피드백 최적화: 병렬 실행(Parallelism)

파이프라인 속도는 생산성과 직결됩니다. 의존성이 없는 테스트들은 병렬로 실행하여 전체 소요 시간을 단축시켜야 합니다.
