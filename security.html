STEP 06

보안 및 데이터 환경 격리

개발 중에 유저 데이터를 날리지 않기 위한 필수적 안전 장치 설계.

보안 설계 원칙 (Pro's Review)

데브옵스에서의 보안은 마지막 단계에 수행하는 '검사'가 아니라, 첫 줄의 코드를 짤 때부터 함께하는 **'문화'**여야 합니다. 이를 **DevSecOps**라고 부릅니다.

이 장에서는 보안 취약점을 자동으로 찾아내고, 개발/검증/운영 환경을 물리적으로 완벽하게 격리하며, 시크릿 정보를 안전하게 관리하는 법을 배웁니다. 최고의 보안은 '막는 것'이 아니라 보안 사고가 나더라도 피해가 확산되지 않는 **격리된 아키텍처**를 설계하는 데 있습니다.

5단계 전문 학습 로드맵

  • 1. Shift-Left Security: 조기 보안 진단

    코드를 짜는 즉시 취약점을 스캔하는 SAST/SCA 도구를 파이프라인에 통합하여 보안 이슈를 미리 제거합니다.

  • 2. Environment Isolation: 물리적 격리

    클라우드 계정과 네트워크 레벨에서 개발(Dev)과 운영(Prod) 환경을 완벽하게 분리하여 "실수로 운영 DB를 삭제하는 일"을 방지합니다.

  • 3. Secrets Management: 시크릿 관리

    .env 파일을 넘어 HashiCorp Vault나 AWS Secrets Manager를 사용해 인증 정보를 안전하게 보관하고 주입합니다.

  • 4. Policy as Code: 규범 준수 자동화

    조직의 보안 규정을 코드로 정의하고(Checkov, OPA), 이를 위반한 인프라는 배포되지 못하도록 강제합니다.

  • 5. Incident Response: 탐지와 대응

    런타임에 일어나는 이상 행위를 감지하고, 장애나 보안 사고 발생 시 빠르게 대응할 수 있는 추적 체계를 구축합니다.
