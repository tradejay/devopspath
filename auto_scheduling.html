Automation Scheduling

귀찮은 반복 작업을 서버가 스스로 정해진 시간에 처리하게 하기

1. 스케줄링(Scheduling)의 개요

모든 자동화가 이벤트 트리거(Webhooks)일 필요는 없습니다. 데이터 백업, 뉴스레터 발송, 이미지 최적화 같은 작업은 정기적으로(예: 매일 자정) 수행되는 것이 좋습니다.

GitHub Actions Cron

별도의 서버를 운영할 필요 없이 GitHub Actions의 `schedule` 이벤트를 사용하면 서버리스 방식으로 주기적인 작업을 실행할 수 있습니다.

2. Cron 표현식 마스터하기

작업 실행 주기를 설정할 때 표준 Cron 문법을 사용합니다.

# ┌───────────── 분 (0 - 59)
# │ ┌─────────── 시 (0 - 23)
# │ │ ┌───────── 일 (1 - 31)
# │ │ │ ┌─────── 월 (1 - 12)
# │ │ │ │ ┌───── 요일 (0 - 6) (일요일=0)
# │ │ │ │ │
# * * * * *

# 사례 1: 매일 새벽 4시에 실행 (0 4 * * *)
# 사례 2: 매주 월요일 정오에 실행 (0 12 * * 1)

3. 주요 자동화 사례

  • Dynamic Sitemap: 매일 새로운 포스트가 추가되었는지 확인하고 사이트맵 재생성.
  • Broken Link Check: 매주 블로그 내부에 깨진 링크가 있는지 전수 조사.
  • Auto Cleanup: 임시 빌드 아티팩트나 30일이 지난 로그 자동 삭제.

실무 조언: 시간대에 주의하세요

대부분의 클라우드 자동화 도구(GitHub Actions 등)는 기본 시간이 **UTC**입니다. 한국 시간(KST)으로 새벽 4시에 실행하고 싶다면 9시간을 뺀 이전 날 오후 7시(19:00 UTC)로 설정해야 함을 명심하세요.
