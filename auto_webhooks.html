Event-driven Webhooks

특정 사건이 발생했을 때 자동으로 작업을 시작하는 트리거 시스템

1. 웹훅(Webhook)의 개념

우리가 서버에 주기적으로 물어보는 것(Polling)이 아니라, 특정 이벤트가 발생했을 때 서버가 우리에게 알려주는(Push) 방식입니다. 자동화 파이프라인의 필수 신경계 역할을 합니다.

주요 활용 사례

  • Content Update: Notion에서 글을 저장하면 즉시 Vercel 배포 트리거.
  • Git Event: 코드를 푸시하면 CI 서버에 빌드 시작 명령 전송.
  • Service Status: 배포가 실패하면 Slack이나 Discord 채널에 메시지 발송.

2. 안전한 웹훅 설계 원칙

웹훅은 공개된 엔드포인트를 노출해야 하므로 보안이 매우 중요합니다.

  • Secret Signature: 요청이 신뢰할 수 있는 소스(예: GitHub)에서 왔는지 헤더의 서명 값으로 검증하세요.
  • Idempotency: 네트워크 오류로 중복 요청이 올 수 있으므로, 동일한 웹훅을 여러 번 받아도 문제가 없게 설계해야 합니다.
  • Async Processing: 웹훅 응답은 즉시(200 OK) 주고, 실제 무거운 작업(빌드 등)은 백그라운드에서 처리하세요.
# Webhook 수신 코드 예시 (Node.js)
app.post('/api/webhook', (req, res) => {
  const signature = req.headers['x-hub-signature-256']
  
  if (verifySignature(req.body, signature)) {
    // 트리거 작업 수행 (예: 빌드 시작)
    console.log("Valid webhook received!")
    res.status(200).send("OK")
  } else {
    res.status(401).send("Unauthorized")
  }
})

실전 팁: 테스트 도구 활용

웹훅을 개발할 때는 로컬 환경에서 테스트하기 어렵습니다. `ngrok`을 사용하여 시각적으로 웹훅 요청이 잘 오는지, 바디 내용은 무엇인지 실시간으로 모니터링하며 개발하세요.
