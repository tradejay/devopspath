서비스 디스커버리 & 네트워킹

Pod IP의 가변성을 넘어서는 Service와 Ingress 아키텍처

1. Pod IP가 죽을 때마다 바뀌는 문제

포드는 생성되고 삭제될 때마다 새로운 IP를 할당받습니다. 따라서 다른 포드에서 해당 포드로 고정된 IP를 통해 통신하는 것은 불가능합니다.

해결책: Service 오브젝트

Service는 여러 포드를 묶어 고정된 DNS 이름과 가상 IP(ClusterIP)를 제공합니다. 개별 포드 IP가 바뀌어도 서비스 이름만 알면 통신이 가능합니다.

2. Service 타입 4가지

타입 용도
ClusterIP 클러스터 내부 통신용 (기본값).
NodePort 모든 노드의 특정 포트를 오픈하여 외부 접근 허용.
LoadBalancer 클라우드 공급자의 로드 밸런서를 사용해 공인 IP 할당.

3. Ingress: 트래픽의 관문

여러 서비스를 하나의 IP로 운영하거나 도메인 기반의 라우팅, SSL 인증서 처리를 하고 싶을 때는 **Ingress**를 사용합니다.

apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: my-ingress
spec:
  rules:
  - host: myapp.example.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: web-service
            port:
              number: 80

안티패턴: NodePort 사용 자제

로컬 테스트용이 아니라면 프로덕션에서 직접 NodePort를 외부에 노출하는 것은 보안상 위험하며 위계적으로도 로드밸런서/인그레스를 통하는 것이 표준입니다.
