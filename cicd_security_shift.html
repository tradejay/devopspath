DevSecOps (Shift-Left)

보안을 배포의 마지막 단계가 아닌, 개발의 시작 단계로 옮기기

1. Shift-Left Security란?

전통적인 방식에서는 배포 직전 혹은 배포 후에 보안 검사를 수행했습니다. 하지만 **Shift-Left**는 개발자가 코드를 작성하는 시점부터 보안 검사를 자동화하여 비용과 리스크를 획기적으로 줄이는 전략입니다.

SAST vs DAST

  • SAST (Static Analysis): 소스 코드를 분석하여 보안 취약점 탐지. (Sonarqube 등)
  • DAST (Dynamic Analysis): 실제 실행 중인 앱에 공격을 시도하여 취약점 탐지. (OWASP ZAP 등)

2. 파이프라인 보안 자동화 항목

  1. Secret Scanning: Git에 API 키나 비밀번호가 평문으로 포함되었는지 검사.
  2. SCA (Software Composition Analysis): 사용 중인 오픈소스 라이브러리에 알려진 취약점(CVE)이 있는지 확인.
  3. Container Scanning: 베이스 이미지나 패키지에 보안 결함이 있는지 스캔. (Trivy 등)
# Trivy를 이용한 취약점 스캔 예시
- name: Security Scan with Trivy
  uses: aquasecurity/trivy-action@master
  with:
    image-ref: 'my-app:${{ github.sha }}'
    format: 'table'
    exit-code: '1' # 심각한 취약점 발견 시 빌드 실패 처리
    severity: 'CRITICAL,HIGH'

조직의 문화: 보안은 모두의 책임

보안 팀이 따로 있는 것이 아니라, 모든 개발자가 자신이 짠 코드의 보안성을 파이프라인 결과를 통해 책임지는 문화를 구축하는 것이 DevSecOps의 궁극적인 목표입니다.
