IaC Foundations (코드형 인프라)

인프라 관리를 위한 가장 현대적인 선언적 철학 이해하기

1. 명령형(Imperative) vs 선언형(Declarative)

인프라를 어떻게 구축하느냐에 따라 두 가지 패러다임으로 나뉩니다.

  • 명령형: "서버 하나 만들고, 태그 달고, 보안 그룹 연결해." 하나씩 명령을 수행. (스크립트/CLI)
  • 선언형: "내가 원하는 최종 상태는 이 보안 그룹을 가진 서버 한 대야." 목표만 설정. (Terraform/K8s)

왜 선언형 IaC인가?

선언형 도구는 실행 시 마다 '현재 상태'와 '원하는 상태'를 비교하여 필요한 부분만 알아서 채워넣습니다(보정 작업). 이는 멱등성(Idempotency)을 보장하여 시스템 안정성을 획기적으로 높여줍니다.

2. 멱등성 (Idempotency)

동일한 인프라 코드를 1번 실행하나 100번 실행하나 결과는 항상 같아야 합니다. IaC 도구는 이를 위해 상태(State) 파일을 관리하며 변경점(Diff)만을 찾아냅니다.

# Terraform HCL (HashiCorp Configuration Language) 예시
resource "aws_instance" "app_server" {
  ami           = "ami-830c94e3"
  instance_type = "t2.micro"

  tags = {
    Name = "DevOpsRoadmapInstance"
  }
}

전문가의 팁: "코드로서의 진실"

IaC를 도입했다면 다시는 웹 콘솔에서 수동으로 버튼을 누르지 마세요. 코드가 실제 인프라의 유일한 진실(Single Source of Truth)이 되어야만 시스템을 완벽히 제어할 수 있습니다.
