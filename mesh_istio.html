Istio 아키텍처 이해

사이드카 프록시를 통한 네트워크 계층의 분리

1. Control Plane vs Data Plane

Istio는 네트워크 트래픽을 처리하는 **Data Plane**과 이를 통제하는 **Control Plane**으로 나뉩니다.

사이드카(Sidecar) 패턴

애플리케이션 컨테이너 옆에 Envoy 프록시 컨테이너를 함께 띄워 모든 입출력 트래픽을 가로채고 제어합니다. 애플리케이션 코드는 네트워크 환경이 바뀐 것을 알지 못합니다.

2. 주요 컴포넌트

  • Istiod: 구성을 관리하고 프록시에 정책을 전달하는 중앙 컨트롤러.
  • Envoy: 초고성능 L7 프록시로, 실제 트래픽을 처리하고 측정 지표를 생성함.
  • Ingress/Egress Gateway: 클러스터 드나드는 트래픽을 위한 전용 프록시 관문.
# 사이드카 자동 주입 설정
kubectl label namespace default istio-injection=enabled

# 주입 확인
kubectl get pods -l app=myapp -o jsonpath='{.items[0].spec.containers[*].name}'
# 결과: myapp, istio-proxy (두 개의 컨테이너가 실행 중이어야 함)

주의: 인지 부하와 리소스 소모

사이드카 프록시는 각 포드마다 메모리를 소모하며 통신 지연(Latency)을 미세하게 발생시킵니다. 시스템 규모가 작다면 서비스 메쉬 도입은 '오버 엔지니어링'이 될 수 있습니다.
