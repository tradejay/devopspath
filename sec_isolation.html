Environment Isolation: 환경 격리

운영 환경을 성역으로 만드는 물리적/논리적 장벽 구축

1. 왜 격리해야 하는가?

개발자가 테스트용 DB를 초기화하려고 실행한 스크립트가 실수로 운영 DB에 연결되어 있었다면? 이런 비극적인 사고를 막는 유일한 방법은 **환경 간의 연결 고리를 물리적으로 끊는 것**입니다.

성숙도 대조표

  • 초급: 하나의 AWS 계정에 'dev-', 'prod-' 이름만 붙여서 같이 씀. (위험)
  • 중급: VPC 레벨에서 네트워크를 분리하고 피어링을 금지함. (양호)
  • 전문가: AWS Account 자체를 Dev용, Prod용으로 따로 만들어 결제와 권한을 완전히 분리함. (최고)

2. 데이터 격리 전략

운영 데이터(Production Data)는 절대 개발 환경으로 흘러들어와서는 안 됩니다. 테스트가 필요하다면 민감 정보를 마스킹(Masking)하거나 익명화한 샘플 데이터만 사용해야 합니다.

3. 네트워크 세분화 (Micro-segmentation)

같은 서버 안에서도 웹 서버가 DB 서버에만 접근할 수 있게 Security Group이나 Network Policy를 좁게 설정해야 합니다. 이것을 **최소 권한의 원칙(Principle of Least Privilege)**이라고 부릅니다.
