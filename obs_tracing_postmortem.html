Distributed Tracing & Post-mortem

서비스 간의 복잡한 요청 흐름을 시각화하고 장애 원인 분석하기

1. 분산 추적(Distributed Tracing)이 필요한 이유

마이크로서비스 아키텍처에서는 하나의 요청이 여러 서비스(A -> B -> C -> DB)를 거칩니다. 특정 구간에서 응답이 느려졌을 때, 어떤 서비스가 병목인지 파악하는 유일한 방법은 전체 흐름을 추적하는 것입니다.

OpenTelemetry (OTel)

벤더에 종속되지 않은 오픈소스 표준 텔레메트리 데이터 수집 규격입니다. 코드 몇 줄로 라이브러리를 추가하면 자동으로 HTTP 요청, DB 쿼리 등에 대한 스팬(Span) 데이터를 생성해줍니다.

2. 트레이싱 데이터 시각화: Jaeger

Jaeger를 통해 수집된 트레이싱 데이터를 간트 차트(Gantt chart) 형태로 시각화하여 다음을 확인할 수 있습니다.

  • Critical Path: 전체 응답 시간에 가장 큰 영향을 주는 구간.
  • Service Dependency: 서비스 간의 복잡한 호출 관계도.
  • Distributed Errors: 어느 서비스에서 에러가 전파되었는지 직관적으로 확인.
# Go 언어에서의 OTel 계측 (Instrumentation) 예시
ctx, span := otp.Tracer("order-service").Start(ctx, "CalculatePrice")
defer span.End()

// 비즈니스 로직 수행
span.SetAttributes(attribute.String("item.id", id))

사후 리뷰(Post-mortem)에서의 활용

장애 발생 후 "왜 이런 일이 생겼나?"를 자문할 때 트레이싱 데이터는 결정적인 증거가 됩니다. 포스트모템 문서에 Jaeger 스크린샷이나 Trace ID를 첨부하여 재발 방지 대책을 구체화하세요.
