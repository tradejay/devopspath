관측성의 3요소 (M.L.T)

시스템의 건강 상태를 입체적으로 분석하는 세 가지 데이터 시각

1. 관측성(Observability)의 핵심 기둥

단순히 서비스가 동작하는지 확인하는 수준을 넘어, 왜 문제가 발생했는지 추론할 수 있는 데이터를 수집해야 합니다.

  • Metrics (지표): 시간 경과에 따른 숫자 데이터. 아그리게이션과 대시보드에 최적화. (예: CPU 사용량, TPS)
  • Logs (기록): 특정 시점에 발생한 이벤트의 상세 내용. 디버깅의 핵심. (예: 에러 스택 트레이스)
  • Traces (추적): 요청 하나가 여러 서비스를 통과하는 여정. 분산 아키텍처의 필수 요소.

왜 세 가지가 모두 필요한가?

지표(Metrics)는 문제가 발생했음을 알려주고(Detection), 추적(Traces)은 어느 구간에서 문제가 생겼는지 짚어주며(Isolation), 로그(Logs)는 구체적으로 어떤 에러인지 알려줍니다(Identification).

2. Modern Observability 아키텍처

최근에는 이 세 가지를 개별적으로 관리하지 않고, 통합된 표준(OpenTelemetry)으로 수집하는 것이 대세입니다.

# 도식화: 데이터의 흐름
[ Application ] --- (Pull/Push) ---> [ Telemetry Collector ]
                                            |
        -----------------------------------------------------
        |                   |                       |
[ Prometheus ]         [ Graylog/Loki ]         [ Jaeger/Tempo ]
  (Metrics)               (Logs)                  (Traces)

실무 조언: 상관관계(Correlation)가 전부다

지표에서 나타난 에러 스파이크를 클릭했을 때 바로 해당 시점의 로그와 트레이스로 연결되지 않는다면, 그것은 진정한 관측성이 아닙니다. 각각의 데이터에 `trace_id`와 같은 고유 식별자를 태깅하세요.
